Types of Keys
  • Super Key
    Any set of attributes that can uniquely identify every row. May have extra attributes.
    R(A,B,C,D): {A,B}, {A,B,C}, {A,B,C,D} are all super keys
  • Candidate Key
    Minimal super key — no proper subset of it is also a super key. A relation can have multiple CKs.
    If {A,B} is a CK, then {A} or {B} alone cannot uniquely identify rows
  • Primary Key
    One candidate key chosen as the main identifier. Cannot have NULL values. Other CKs become alternate keys.
  • Foreign Key
    References the primary key of another relation. Enforces referential integrity. Can contain NULLs.
    R.A references S.A (R = child/referencing, S = parent/referenced)
Key Rule: Every CK is a super key, but NOT every super key is a CK. An attribute that never appears on the RHS of any FD MUST be part of every candidate key.
Functional Dependencies

X → Y means: any two tuples with the same X values must have the same Y values.

Trivial vs Non-trivial
  • Trivial FD
    X → Y where Y ⊆ X. Always holds. E.g., AB → A, ABC → BC
  • Non-trivial FD
    X → Y where Y ⊄ X. E.g., A → B, AB → CD. These are useful for normalization.
Armstrong's Axioms (Sound and Complete)
1. Reflexivity: Y ⊆ X ⟹ X → Y 2. Augmentation: X → Y ⟹ XZ → YZ 3. Transitivity: X→Y, Y→Z ⟹ X → Z ── Derived Rules ────────────────────── 4. Union: X→Y, X→Z ⟹ X → YZ 5. Decomposition: X → YZ ⟹ X→Y and X→Z 6. Pseudotransitivity: X→Y, WY→Z ⟹ WX → Z
Finding Attribute Closure X⁺ — used to check if X is a candidate key

Start with X⁺ = {X}. Repeatedly apply FDs to expand until no further expansion is possible.

R(A,B,C,D,E), FDs: A→BC, CD→E, B→D, E→A Compute A⁺: Start: {A} A→BC: {A, B, C} B→D: {A, B, C, D} CD→E: {A, B, C, D, E} ← all attributes covered ∴ A⁺ = {A,B,C,D,E} → A is a Candidate Key ✓
Exam Strategy: To find ALL candidate keys, compute closure for every attribute and every combination. If closure = all attributes, it's a candidate key. If a strict subset also gives all attributes, then it's not minimal — discard it.

Key Definitions for Normalization
  • Prime Attr
    An attribute that belongs to at least one candidate key.
  • Non-Prime Attr
    An attribute that does NOT appear in any candidate key.
  • Partial Dep
    A non-prime attribute depends on only a PART of a composite candidate key.
    R(A,B,C,D); CK={A,B}; if A→C then C partially depends on {A,B}
  • Transitive Dep
    A non-prime attr A determines another non-prime attr B: CK → A → B (indirect dependency).
    EmpID → DeptID → DeptName : DeptName has transitive dependency on EmpID
BCNF vs 3NF — Most Tested Difference
3NF: X → Y is allowed if: (a) X is a super key, OR (b) Y is a prime attribute (part of some CK) BCNF: X → Y is allowed ONLY if: (a) X is a super key (no exceptions!)
R(A, B, C), FDs: AB→C, C→B Candidate Keys: {A,B} and {A,C} Prime attributes: A, B, C (ALL are prime here!) Check FD C → B: Is C a super key? NO (C⁺ = {B,C} ≠ all) Is B a prime attribute? YES (B ∈ CK {A,B}) Conclusion: In 3NF ✓ but NOT in BCNF ✗ (This exact example appears repeatedly in UGC NET!)
Preservation: BCNF guarantees lossless-join decomposition but may NOT preserve all FDs. 3NF guarantees BOTH lossless-join AND dependency preservation. This distinction is asked very frequently in UGC NET.
Quick Decision Rule for Any FD X→Y
For each non-trivial FD X → Y: │ ├── Is X a SUPER KEY? │ ├── YES → BCNF ✓ (also 3NF ✓) │ └── NO → │ └── Is every attr in Y a PRIME attribute? │ ├── YES → 3NF ✓ (NOT in BCNF) │ └── NO → Not even in 3NF ✗

Division Operation — "for all" queries

R ÷ S gives tuples in R that are associated with ALL tuples in S. Used for queries like "find students who enrolled in ALL courses."

R(Student, Course), S(Course) = {DBMS, OS, CN} R ÷ S = students who took ALL three courses Equivalent expression: R ÷ S = π_x(R) − π_x( (π_x(R) × S) − R ) where x = attributes in R not in S = {Student}
Tuple Count Formulas
R has m tuples, S has n tuples: • R × S → exactly m × n tuples • R ∪ S → max(m,n) to m+n tuples • R ∩ S → 0 to min(m,n) tuples • R − S → max(0, m−n) to m tuples • R ⋈ S (Natural Join) → 0 to m×n tuples • σ_cond(R) → 0 to m tuples
SQL Execution Order
1. FROM → Identify and load tables 2. JOIN → Combine tables (if any) 3. WHERE → Filter individual rows (no aggregates here!) 4. GROUP BY → Group rows by specified columns 5. HAVING → Filter groups (aggregates allowed here) 6. SELECT → Pick columns, apply functions/aliases 7. ORDER BY → Sort the results 8. LIMIT → Restrict number of output rows
Critical Exam Point: WHERE executes BEFORE GROUP BY — you cannot use aggregate functions (COUNT, SUM, AVG) in WHERE. Use HAVING to filter aggregated results. WHERE filters rows; HAVING filters groups.
SQL Joins
INNER JOIN → Only matching rows from both tables LEFT OUTER JOIN → ALL from left + NULLs for non-matches in right RIGHT OUTER JOIN → ALL from right + NULLs for non-matches in left FULL OUTER JOIN → ALL rows from both (NULLs where no match) CROSS JOIN → Cartesian product (no condition) NATURAL JOIN → Auto-joins on column(s) with same name

Concurrency Problems
  • Dirty Read
    T1 reads uncommitted data from T2. If T2 aborts, T1 has used invalid data.
  • Lost Update
    T1 and T2 both read X, update it. T2's write overwrites T1's update — T1's work is lost.
  • Non-repeatable Read
    T1 reads same row twice; T2 modifies it in between → T1 gets different values on second read.
  • Phantom Read
    T1 runs same query twice; T2 inserts/deletes matching rows in between → T1 sees different set of rows (phantom rows).
Serializability
  • Conflict
    Two operations conflict if: from different transactions, on same data item, and at least one is WRITE.
    R-W, W-R, W-W conflict. R-R does NOT conflict.
  • Conflict Serializable
    Schedule can be transformed to a serial schedule by swapping non-conflicting adjacent operations.
  • Precedence Graph
    Draw edge Ti→Tj if Ti has a conflicting op before Tj on the same data item.
    Schedule is conflict serializable ⟺ Precedence graph is acyclic (DAG)
  • View Serializable
    Broader than conflict. Every conflict-serializable schedule is view-serializable, but NOT vice versa. Checking is NP-complete.
Hierarchy: Conflict Serializable ⊂ View Serializable ⊂ All Serializable schedules. In practice, only conflict serializability is checked (via precedence graph). Acyclic = serializable.
Two-Phase Locking (2PL)
Phase 1 (Growing): Only ACQUIRE locks — no releasing allowed Phase 2 (Shrinking): Only RELEASE locks — no acquiring allowed Lock Point = moment of last lock acquisition Variants: • Basic 2PL: May cause cascading rollbacks • Strict 2PL: Hold WRITE locks until commit/abort • Rigorous 2PL: Hold ALL locks until commit/abort (prevents cascading rollbacks) Lock compatibility: Shared (S) lock ←→ Read operation Exclusive (X) lock ←→ Write operation S-S: Compatible ✓ S-X: Incompatible ✗ X-X: Incompatible ✗
2PL guarantees conflict serializability but can cause DEADLOCKS. Deadlock handling: prevention (timestamps), detection (wait-for graph), or avoidance.

Generalization, Specialization, Aggregation
  • Specialization
    Top-down. Split a general entity into subclasses with additional unique attributes.
    Employee → Manager, Technician, Clerk
  • Generalization
    Bottom-up. Combine similar entities sharing common attributes into a superclass.
    Car + Truck + Bus → Vehicle
  • Aggregation
    Treat a relationship set as a higher-level entity when it must participate in another relationship.
    Used when a relationship itself has relationships with other entities

B-Tree vs B+ Tree
FeatureB-TreeB+ Tree
Data pointersAll nodes (internal + leaf)Leaf nodes ONLY
Leaf linkingNot linkedLinked list (doubly linked)
Range queriesInefficient — requires tree traversalVery efficient — scan linked leaves
Search resultMay end at internal nodeAlways reaches leaf node
HeightShorterSlightly taller
DBMS usageFile systemsAll major DBMS (MySQL, PostgreSQL)
B+ Tree of order m: • Internal node: min ⌈m/2⌉ children, max m children • Leaf node: min ⌊(m-1)/2⌋ keys, max m-1 keys • Root node: min 2 children (unless it's also a leaf) • All leaves at the SAME LEVEL (perfectly balanced)
B+ Trees are the standard for DBMS indexes. Internal nodes store only keys for routing; leaf nodes store key + record pointer pairs. The linked leaf list enables efficient ORDER BY and range queries.
Hashing
  • Static Hashing
    Fixed number of buckets. h(key) = key mod n. Overflow via chaining (linked overflow buckets).
    Problem: inefficient if data grows significantly beyond bucket capacity
  • Extendible Hashing
    Dynamic. Uses a directory that doubles when a bucket splits. Global depth vs Local depth.
  • Linear Hashing
    Splits buckets in a linear, predetermined order without needing a directory. Gradual growth.
Hashing vs B+ Tree: Hashing → O(1) for exact-match searches, but POOR for range queries. B+ Tree → O(log n) but excellent for range queries and ORDER BY. DBMS uses B+ Trees for indexes; hashing for hash-joins.
